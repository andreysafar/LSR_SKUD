# Анализ готовности LSR_SKUD vs Техническое задание

**Дата анализа:** 2026-03-25
**Оценка готовности:** ~25-30%

## Цель проекта (по ТЗ)

Организация пропускного режима ЖК: считывание номеров, разовые/гостевые пропуска, журнал въезда/выезда, контроль меток, чёрный список, уведомления. Пресечение торговли метками и незаконной сдачи м/м. Три роли: Собственник, Охрана, УК.

---

## Что реализовано (работает)

1. **Распознавание номеров** — YOLO26n + EasyOCR, live-камеры и batch-обработка на GPU
2. **Telegram-бот** — авторизация по телефону через Parsec, создание vehicle pass (3ч/24ч/неделя/до конца дня)
3. **Интеграция с Parsec СКУД** — SOAP API: сессии, поиск людей/т/с, создание идентификаторов, открытие шлагбаума
4. **Автоматическое открытие шлагбаума** — при распознавании номера с активным пропуском
5. **Логирование gate_events** — запись событий шлагбаума (но не парный журнал)
6. **Streamlit-дашборд** — камеры, события распознавания, настройки, batch-аналитика
7. **Docker** — контейнеризация с GPU (nvidia runtime), multi-stage build
8. **Batch-обработка видео** — dual-pipeline (FFmpeg CPU + Neural GPU), до 1000 файлов/час
9. **Тренировка моделей** — сбор обучающих данных через admin-review в Telegram

## Оптимизация GPU (видеоаналитика)

### Текущее состояние
- **Модели:** YOLO26n (nano) — быстрый, но может терять точность на мелких номерах
- **OCR:** EasyOCR — **узкое место**, последовательная обработка по одному номеру
- **Оптимизация моделей:** TorchScript export с FP16 — работает
- **TensorRT:** НЕ используется (потенциальное ускорение 2-5x)
- **Батчинг:** НЕ реализован (single-frame inference)
- **Трекинг номеров между кадрами:** НЕТ (нет temporal consistency)
- **Определение направления (въезд/выезд):** НЕТ (камеры не размечены по направлению)
- **Multi-GPU:** поддержка через CUDA_VISIBLE_DEVICES, но нет load balancing для live

### Необходимые оптимизации (Фаза 3)
1. **TensorRT export** — замена TorchScript на TensorRT для inference
2. **Батчинг кадров** — несколько кадров за один inference call
3. **Замена/дополнение EasyOCR** — PaddleOCR или специализированный OCR для номеров РФ
4. **Трекинг номеров** между кадрами для надёжности и определения направления
5. **Разметка камер по направлению** — entry/exit для парного журнала

## План реализации (5 фаз)

### Фаза 1: Ядро системы — БД, пропуска, журнал

**1.1. Расширение схемы БД**
- Таблица `owners` — собственники (ФИО, кв., тел., кол-во м/м)
- Таблица `trusted_persons` — доверенные лица собственников
- Таблица `parking_spots` — машиноместа (номер, собственник)
- Таблица `blacklist` — чёрный список (собственник, причина, дата)
- Таблица `incidents` — инциденты (тип, участники, описание, создатель)
- Таблица `entry_exit_log` — парный журнал въезда/выезда
- Расширение `passes`: типы `loading` (разовый 40 мин) и `guest` (гостевой с м/м)
- Расширение `cameras`: поле `direction` (entry/exit)

**1.2. Новые типы пропусков в боте**
- Разовый на погрузку/разгрузку (автоматически 40 мин)
- Гостевой с указанием м/м (валидация, что м/м принадлежит собственнику)

**1.3. Парный журнал въезда/выезда**
- Привязка камер к направлению
- Запись entry → автоматический поиск exit → расчёт длительности

### Фаза 2: Роли, уведомления, ЧС

**2.1. Ролевая модель**
- Роль "Собственник" (текущий + расширение)
- Роль "Охрана" (рабочее место в Telegram)
- Роль "УК" (админ-функции)
- Роль "Доверенное лицо" (пропуска от имени собственника)

**2.2. Система уведомлений (APScheduler)**
- За 5-10 мин до окончания разового пропуска → собственнику
- Прибытие/убытие гостя → собственнику
- Превышение 40 мин разовым → охране + собственнику
- Въезд без пропуска/метки → охране + УК
- Несоответствие номера и метки → охране

**2.3. Чёрный список и инциденты**
- CRUD для ЧС (добавление/удаление/просмотр)
- Авто-блокировка при повторном нарушении (1-й раз → предупреждение, 2-й → ЧС)
- Проверка ЧС при создании пропуска (отказ + уведомление)
- Фиксация инцидентов охраной (тип, участники, описание)

### Фаза 3: GPU-оптимизация

**3.1. TensorRT**
- Export YOLO моделей в TensorRT (.engine)
- Fallback на TorchScript

**3.2. Определение въезд/выезд**
- Поле `direction` в конфиге камер
- Логика парного журнала в pipeline

**3.3. OCR-оптимизация**
- Батчинг табличек
- Оценка PaddleOCR как альтернативы

### Фаза 4: Контроль меток (через Parsec)

**4.1. Интеграция меток**
- Получение идентификаторов (меток) из Parsec API
- Кэширование привязок метка↔номер т/с
- Проверка соответствия при въезде по метке

**4.2. Лимит одновременного нахождения**
- Подсчёт т/с собственника на территории через entry_exit_log
- Если т/с ≥ м/м → отказ в допуске (метка не открывает)

**4.3. Рабочее место охраны (Telegram)**
- Команды: список пропусков, создание пропуска, журнал, инциденты
- Inline-уведомления о нарушениях

### Фаза 5: УК и отчёты

**5.1. Админ-панель УК (Streamlit)**
- CRUD: собственники, м/м, метки
- Управление ЧС
- Просмотр/управление пропусками

**5.2. Экспорт отчётов**
- CSV/Excel: реестры пропусков за период, ЧС, инциденты, несоответствия меток
- Фильтрация по дате, типу, собственнику

---

## Решения

- **Порядок реализации:** последовательно Фаза 1 → 2 → 3 → 4 → 5
- **Мессенджер Макс:** отложен, фокус на Telegram
- **Метки:** управляются через Parsec (идентификаторы), нужна логика контроля соответствия и лимита
